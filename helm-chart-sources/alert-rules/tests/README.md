# Unit-тесты alert-rules


## Структура

| Файл | Назначение |
|---|---|
| `run.sh` | Рендерит **весь** чарт (все группы из всех файлов правил) во временную папку и прогоняет против него все `*_test.yaml` из этой директории (использует `promtool`, либо `prom/prometheus` через docker как фоллбэк). |
| `*_test.yaml` | Кейсы `promtool test rules`. Один файл на правило (или на группу связанных правил). |

Отрендеренные правила — временный артефакт: `run.sh` генерирует их во временную
директорию на время прогона и удаляет после, так что в репозиторий ничего лишнего
не коммитится, а тесты всегда соответствуют актуальным выражениям чарта.

Имена правил/групп/файлов нигде не захардкожены: `run.sh` собирает values из
`files/defaults/*.yaml` и принудительно включает все `enabled`, затем рендерит
чарт целиком. Поэтому тест на любое правило — в любом файле, текущем или будущем
(например для троттлинга) — работает без правок скрипта.

## Запуск

```bash
./run.sh
```

## Как добавить тест на другое правило

1. Создай `<что-нибудь>_test.yaml` в этой директории (например `in_cloud_capi_cluster_ready_test.yaml`).
2. Укажи в `rule_files` общий отрендеренный файл:

   ```yaml
   rule_files:
     - rendered.rules.yaml
   ```

3. Опиши свои `input_series` и кейсы `promql_expr_test` / `alert_rule_test`,
   затем обращайся к recording-метрике по имени (например `in_cloud_capi_cluster_ready`).
4. Запусти `./run.sh` — он сам подхватит новый файл. Править скрипты не нужно.

`run.sh` рендерит **весь** чарт, поэтому тест может ссылаться на любое правило из
любого файла — включая цепочки (например `in_cloud_capi_cluster_ready` зависит от
`in_cloud_capi_cluster_nodes_consistent`, который зависит от правил
`*_consistent`). Правила, чьи входные данные тест не подаёт, вычисляются в пустоту
и не мешают.

### Единственный нюанс: input_series против recording-правил

Часть имён метрик в группе — это сами recording-правила (например
`in_cloud_capi_cluster_deletion_requested`, правила `*_consistent` /
`*_update_required`). Для каждого правила есть два способа подачи данных:

- **Подать recording-метрику напрямую** как `input_series` (проще всего, удобно
  для изоляции тестируемого правила). Безопасно, **пока ты не подаёшь
  одновременно исходные метрики**, из которых recording-правило вычисляет
  значение — иначе правило пересчитает метрику и затрёт твои сэмплы. Mute-тест
  подаёт `in_cloud_capi_cluster_deletion_requested` напрямую и никогда не подаёт
  clusterclaim-метрики, из которых она выводится, поэтому recording-правило
  остаётся пассивным.
- **Подать листовые исходные метрики** и дать recording-правилу самому вычислить
  значение (проверяет реальную композицию end-to-end). Используй, когда
  тестируешь именно само recording-правило.

Не делай и то, и другое для одной метрики в одном тесте.
